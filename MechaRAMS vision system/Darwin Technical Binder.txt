Software – Idun
Idun
Darwin eliminates the performance constraints of the roboRIO by oﬄoading 
almost all processing to its onboard M4 Mac mini. This enables low-latency loop 
cycles at 200 Hz, highly eﬃcient log compression, and robot code that can be 
trusted to always perform at its best; no loop overruns, memory related crashes, 
or other reliability issues caused by the roboRIO’s poor performance.
Fun Fact: Darwin is named after the operating system that runs its robot code!
Architecture
Features 
•Auto-generated code minimizes boilerplate 
•Annotation processors generate code based 
on IO interfaces
•Constants are shared between robot programs
•Network protocols enable low-latency 
communication 
•Input and output data streamed via a custom 
UDP protocol running over Ethernet
•Safety interlocks and fallback modes 
•Idun protect Darwin and anyone around the 
robot in all circumstances
•Fallback modes enable basic robot 
functionality even during major faults
Darwin runs C++ code on the roboRIO to interface with hardware, while the Java robot code runs on the 
Mac mini. AdvantageKit’s IO interfaces provide a clean abstraction layer.

•Real-time communication enables eﬃcient 
on-ﬁeld debugging
•Every device logs ping status, latency, and 
more for deep diagnostics
•200 Hz logs are compressed using XZ 
•Compression factor of ~3-5, enabling smaller 
sizes than 50 Hz roboRIO logs
•Highly-optimized Java runtime 
•Runs on a Java 25 JRE with generational 
ZGC and compact object headers
•Thread policies in the Mach kernel enable 
near-RT performance in macOS

Software – Controls
Launching on the Move
Flywheel Control
Darwin is designed for launching on the move:
•Position & velocity is measured using AprilTag 
localization on four cameras.
•Recursive time-of-ﬂight lookup are augmented 
with a model to compensate for linear drag.
•Feedforward models enable 200 Hz target tracking 
with under 0.5° drive error while in motion.
•Joysticks controls are dynamically limited based on 
the polar velocity relative to the hub, maximizing 
both speed and accuracy.
•Auto-alignment uses a dynamic center-of-rotation 
to minimize launcher acceleration while maintaining 
intuitive controls for the driver.
See the code here.
Darwin’s ﬂywheel automatically switches between control modes. Open-loop control is used when idle to 
minimize current draw. A custom motion proﬁle generates intermediate setpoints during spin-up to 
maintain constant supply current, adjusting dynamically based on the match phase to avoid brownout.
PID + Feedforward 
Used for close volleys and passing. 
•Feedforward model using KS, KV, KA
•1000 Hz velocity PID controller
•Slow ~20 ms recovery from each launch, 
reducing accuracy
•Total supply current draw of 80-100A while 
actively launching
•Balances accuracy and current draw 
requirements for trajectories that require 
minimal precision
Dynamic Bang-Bang 
Used for far volleys (e.g. outpost). 
•200 Hz bang-bang controller running:
•High “boost” voltage (below setpoint)
•Feedforward voltage (above setpoint)
•Near-instant recovery after every launch for 
maximum accuracy
•Total supply current draw of 120-160A while 
actively launching
•Aggressive control mode that drains battery 
and must be used sparingly
See the code here.

Software – Autos
Path Following 
Darwin’s autos combine time-optimal pre-
generated trajectories with dynamic paths 
generated based on localization data.
•Alliance zone paths and neutral zone sweeps use 
time-parameterized trajectories for maximum 
control.
•Paths returning from the neutral zone use 
position-parameterized trajectories to handle 
unexpected disturbances.
See the code here.
Anti-Beaching 
During auto, Darwin applies a corrective velocity to 
prevent beaching as soon as the robot begins to 
ride up on a fuel: a critical feature for rough 
collisions with opponents in the neutral zone. 
Darwin’s 200 Hz loop enables a sub-5 ms reaction 
time, countering the robot’s momentum before it 
can ride further up a pile of fuel.
See the code here.
Vehicle Trajectory Service
Time-parameterized trajectories are generated 
using our custom Vehicle Trajectory Service. 
Every waypoint and constraint is deﬁned in 
code and generated automatically at build time 
by the Choreo CLI.
Path segments can be combined in arbitrary 
orders programmatically, enabling hundreds of 
unique combinations without requiring 
individual tuning for each path (important for 
maximizing compatibility with partners).
See the code here.

Salesman Solver 
Darwin supports dynamic intaking in auto based on fuel localization, utilizing the performance of the M4 
Mac mini to process hundreds of game pieces and millions of potential paths in real-time.
•Hundreds of fuel are fully localized within a 230° FOV around the front of the robot using 3 global-
shutter cameras and 12 parallel object detection pipelines.
•Real-time model predictive control (MPC) using Sleipnir calculates the best path through the fuel on-
the-ﬂy, optimizing for the expected fuel count within time and positions constraints.
•Real-time robot detection enables strategic fuel poaching on the opposing side of the ﬁeld (prior to 
Team Update 22) by adjusting position bounds dynamically.
Example: Darwin’s Benevolent Salesman auto used an early version of the salesman solver to 
continuously ferry fuel from the neutral zone to the alliance zone during auto.

Software – Energy
Energy
Current draw from every subsystem and control system 
component is logged and integrated at 200 Hz to provide 
detailed energy statistics. Darwin’s total energy usage 
was 12.8 kWh!
Darwin models the battery chemistry and breaker 
dynamics to predict the exact allowable current draw and 
dynamically adjust current limits to maximize 
performance without browning out.
Finance Department 
The “ﬁnance department” allocates current to the drive based on the output of the battery estimator, 
allowing the robot to safely utilize the full potential of the battery in every match.
Battery Estimator 
•Coulomb counting with a dynamic Peukert’s 
correction estimates battery State of Charge 
(SOC) and models transient voltage sag 
through a single RC Thevenin model1.
•Measured voltage is compared to the model 
(based on empirical Open Circuit Voltage and 
internal resistance curves) to update the 
estimator state using a noise-attenuated 
Kalman-style gain.

Prototyping – Dye Rotor Indexer
Dye Rotor Indexer
The goal of this prototype was to explore the viability of a dye-rotor feeder system for quickly indexing 
fuel into a single-stream launcher, such as a turret.
Iteration #1 Iteration #2
We determined that it was simplest to rotate the 
drum instead of the central tower, removing the 
need for complex gear systems. We created a 
cardboard-and-tape version. The drum was 24” 
in diameter, approaching the maximum size 
allowed by robot frame perimeter requirements.
Moved the feeder to the center of the rotor, 
added 3D printed “blades” to agitate fuel on the 
ﬂoor, decreased height to expand hopper. The 
20” drum was driven by a 3” compliant wheel 
that allowed the drum to slip if jams occurred. 
0.125” of compression in the kicker ramp.
Key Learnings 
•Jams are frequent with this architecture, 
especially when the fuel feeder is placed 
oﬀ-center.
•More friction/agitation was required 
between the fuel and the rotating drum to 
guide them towards the output.
•Final results: 
•Output fuel per second: 1-2
•Capacity: 16-20 fuel
•Indexing time: 1.0-2.0 sec
Key Learnings 
•Use smooth surfaces to prevent fuel from 
being destroyed on sharp corners.
•The blades should be taller (~4”) to funnel 
fuel instead of passing through.
•Add more powered kickers along the 
blades to allow for faster drum speed.
•Final results: 
•Output fuel per second: 4-6
•Capacity: 40-50 fuel
•Indexing time: 0.5-1.0 sec

Outcome 
Eventually tabled due to the move from a single-stream to multi-stream robot architecture.

Prototyping – Cassette Indexer
Cassette Indexer
Same-direction omni wheels feeding a single-stream output. Successfully achieved high fuel per second 
but was tabled when switching from a turret to drum architecture.
Iteration #1: Initial Design Iteration #2: Refinement
Reused much of the Twindexer’s geometry and 
compression. Both omni wheels spun the same 
direction, with two supplementary compliant-
wheel rollers and a kicker at the exit.
Testing clips: Video 1, Video 2, Video 3
•Cones on the omni wheels to guide incoming 
fuel onto the wheel surface
•Roof on the exit to avoid jamming 
•Guard (“fairy house”) to protect the dead zone 
between the wheels
•Second motor on the main wheels
Testing clips: Video 1, Video 2
Key Learnings 
Promising at low fuel volumes, single-stream 
output worked. But at full hopper, the weight 
of the fuel caused jams and prevented 
organized ﬂow into the indexer.
Key Learnings 
The fairy house and cones eliminated the worst 
dead spots, the second motor unlocked higher 
output speeds, and the prototype set up the 
Twindexer team for higher-ﬁdelity testing on 
the same chassis.
Outcome 
Ultimately tabled when the overall robot architecture moved from a turret to a drum launcher. The 
prototype provided key learnings for both the cassette concept and the Twindexer, whose group later 
repurposed this build for higher-ﬁdelity tests.

Prototyping – Spindexer
Spindexer
We started our Spindexer development with a version utilizing “ﬁns” and transitioned to a single omni 
wheel in the center, experimenting with active ﬂoors, diﬀerent-sized wheels, and exit geometry. In the 
end, the prototype was deprecated due to its incompatibility with our shifting robot architecture.
Iteration #1 Iteration #2
Inspired by Team 1986’s 2017 design, we built 
a rough prototype with polycarb ﬁns. However, 
we experienced numerous issues such as:
•Loss of momentum upon contacting fuel
•Jams under high volumes
•Poor durability from the ﬁns themselves
Testing video here.
V1 Spindexer CAD
After seeing a handful of simple single-wheel 
designs we decided that a design without 
dividers would reduce jamming risks.
We started with a four-inch Colson wheel with 
1/16” polycarb sides in a wooden hopper, 
pictured above.
The biggest failure points were getting fuel to 
consistently fall into the circle and to exit 
without jamming. We concluded that the exit’s 
width had to be perfect in order to ﬁt exactly 
one fuel (any wider and two would wedge 
together) and that we should be powering the 
exit to help fuel out faster.
With these issues in mind, we made some 
minor adjustments to the geometry of the exit, 
swapped the colson for a six inch omni wheel, 
and powered the exit with compliant wheels.
Testing clips: Video 1, Video 2

Prototyping – Spindexer
Iteration #3
Outcome
Identifying build quality as our main 
throughput bottleneck, we moved onto a 
higher-ﬁdelity version. We designed a funnel 
out of 16” polycarb, powered the system 
without hand drills, and, trying to more 
realistically replicate the size of our actual 
robot, we designed a 10” omni wheel for this 
iteration.
After some quick tests we were still unhappy 
with the fuel per second. The exit geometry 
was still a problem, so we relocated the exit 
and adjusted the prototype to be more of a 
whistle shape, hoping the fuel would carry its 
momentum better.
In an eﬀort to create a better ﬂow of fuel, we 
also experimented with adding an active ﬂoor 
that spun with the same shaft as the omni 
wheel and a cone on top of the omni.
Testing clips: Video 1, Video 2
This design was set aside following a strategic 
shift from a turret to a multi-lane launcher 
architecture. However, it provided knowledge 
that we applied to its successor, the 
Twindexer.
Key Learnings 
•Exit geometry is crucial 
•The width of the exit needs to 
accommodate one fuel loosely enough to 
make exiting easy, but tight enough to 
prevent two fuel from jamming.
•Create a tangential exit if possible to 
guide momentum instead of making the 
fuel turn a sharp corner to exit.
•Power the exit.
•Maximize gravity feeding 
•No internal dividers, high volumes will 
create jams.
•Omni wheels and a cone on top help fuel 
fall into the circle.

Prototyping – Tunnel Indexer
Tunnel Indexer
Belt-driven 118-style indexer, four iterations chasing reliable hopper clearance before tabling. The goal 
was to reliably meter and transport a full hopper of fuel into a single stream at maximum speed and 
consistency.
Iteration #1: Low-Fidelity Iteration #2: High-Fidelity
Inspired by FRC Team 118’s 2017 indexing 
system. Built quickly to validate the core belt-
driven transport concept before committing to a 
higher-ﬁdelity build. Issues found and ﬁxed in-
place:
•Belt spacing & sag: moved belts closer 
together and added a ﬂoor
•Belt slippage: designed adjustable bearing 
holders to tension belts
•Tunnel backspin: added a belt at the top to 
create 0.25” of compression
Videos of tunnel backspin and resolution.
Key Learnings 
Repeatedly self-destructed under load (belts 
slipped oﬀ pulleys, bearings popped out, 
etc). Served its purpose as a proof-of-
concept; observing real hopper-clearing 
speed and consistency would require a 
substantially higher-ﬁdelity rebuild.

Prototyping – Belt Floor Indexer
Belt Floor Indexer
Our 254-style V indexer (belt ﬂoor indexer) grew out of our transition to a multi-lane launcher; we were 
curious to see how our belt-based tunnel indexer would work without the tunnel. A recurring observation 
throughout both prototypes was just how critically belt spacing and tensioning aﬀect performance. Too 
much or too little sag means the fuel doesn't move through the hopper eﬀectively, and that hiccup at 
step one propagates through the entire system.
Initial Design
The initial design focused on outputting into 
two separate lanes, repurposing much of the 
earlier belt hopper architecture but now 
feeding directly into a multi-lane launcher 
rather than a tunnel. Early tests indicated we 
needed to circle back and increase the ﬁdelity 
of the system before drawing ﬁrm 
conclusions.
Key Learnings 
• Lane splitting slows things down 
•We initially focused on outputting into two 
separate lanes, then removed the divider to 
improve throughput.
•Side-to-side indexing caused belt failures 
•Belts consistently jammed or slipped oﬀ due 
to the force of the fuel. Notably, this was 
before we redesigned to a full-width 
launcher, and results may have been 
diﬀerent without the side-to-side indexing 
requirement.
•Belts need maximum grip 
•Whether by sewing nubbins on or coating 
them, the grip the belts have on the fuel is 
crucial for propelling fuel properly and 
reducing backspin.
•Friction is critical 
•Any and all ways to reduce friction (getting 
center-to-center distances right, etc.) are 
essential for consistent performance.

Prototyping – Hot Dog Indexer
Hot Dog Indexer
Because we were trying to funnel the balls into a shooter that wasn’t full-width yet, belts weren’t going to 
cut it due to the horizontal force of the balls taking them clean oﬀ the pulleys. Instead, we transitioned to 
a set of rollers leading into the two vertical rollers we had on the belt ﬂoor prototype.
Initial Design
We experimented with the conﬁguration of top 
rollers, including wheels and stars. The 
conﬁguration of the rollers did not signiﬁcantly 
aﬀect indexing, so we focused on reducing 
jams through agitating the fuel and reducing 
complexity. We ended up with just one shaft 
with star wheels.
Key Learnings 
•Lots of friction = bad
•Reduced center-to-center distance by 
0.040” to reduce friction
•Grippy rollers are important! We used grippy 
tape on the rollers for the prototype and 
silicone on the comp bot.
•Fast = important
•We don’t want our shooting to be limited by 
our indexing so everything was run as fast 
as possible (4:1, eventually 1.5:1).
•1.5” polycarbonate rollers work well
•AndyMark stealth wheels 2” 35A
•Sticky wheels are important so that the fuel 
does not bounce oﬀ
•Top roller didn’t contribute signiﬁcantly
•We wanted to make sure we’re impeding 
maximum ball ﬂow as little as possible, so the 
stealth wheels were spaced out as wide as 
possible.

Prototyping – Climber
Climber
At the start of the season, we prioritized climbing, aiming for a complete climb in 3–5 seconds to make it 
strategically valuable. Our ﬁnal design was inspired by FRC 3255 The SuperNURDS' one-stage dual-
carriage elevator, adapted to reach 30 inches. By mechanically linking the carriages with chains, one 
carriage could always reach for the next rung while the other lifted the robot, maximizing climb speed 
and eﬃciency.

Key Learnings 
•Latch friction as it slid was critical to climb reliability. We reduced friction using four 0.25" ID, 0.5" 
OD bearings on each latch.
•Maintaining chain tension was diﬃcult, as tension varied throughout the chain. Even with a 
turnbuckle, achieving consistent tension proved challenging.
•As each tower level passed the bumpers, interference occurred at the top of the bumper. We 
proposed a deployable ramp attached to the elevator to create a smoother transition and reduce 
contact.
•Latch deployment remained an unsolved challenge. Our proposed solution used spring-loaded 
latches held by magnets and connected to strings anchored on the elevator. Elevator movement 
would tension the strings, disengage the magnets, and release the latches for deployment.
Outcome 
The climber was tabled on the competition robot based on a strategic assessment after week 0/1 
competitions. It would have added ~14 lbs to the robot, which was more beneﬁcial when allocated to 
other subsystems. Ultimately, the space dedicated to the climber on the week 0/1/3 robot was used 
when transitioning to a full-width launcher.
